# 我把 Hermes 装进了 iPhone：一个支持实时语音对话的开源 PWA

平时使用 Hermes 时，我一直有一个很直接的需求：能不能不坐在电脑前，也不用打开终端，只拿起 iPhone 说句话，就让运行在自己服务器上的 Hermes 帮我处理事情？

比如询问之前保存的资料、梳理当天安排，或者在开车和做家务时继续一段对话。输入应该是语音，回复也应该直接播放出来，而且最好像正常交谈一样，不需要每说完一句都再按一次按钮。

围绕这个需求，我做了 Call Hermes。

它不是一个套壳聊天页面，而是一条完整的实时语音链路：iPhone 负责采集和播放音频，服务端负责语音识别、调用 Hermes、合成语音，再通过 WebRTC 把声音送回手机。

虽然项目最初从 iPhone 使用场景出发，但它并不局限于 iPhone。PC 端 Chrome、Edge 和 Android Chrome 同样可以使用主要的实时通话、设备选择与 Debug 功能；只是当前测试和交互优化仍以 iPhone Safari 为重点。

项目已经开源：

https://github.com/ronnnhui/call-hermes

## 为什么最后选择 PWA

最初的设想是开发一款原生 iPhone App。但对于个人项目来说，原生 App 意味着 Xcode、签名、证书、安装和版本发布，调试链路也更长。

PWA 更适合第一阶段验证：在现代浏览器中打开网页并授权麦克风，就可以直接使用；在 iPhone Safari、Android Chrome 或桌面 Chrome/Edge 中还可以安装为 PWA。前端仍然可以调用 WebRTC、麦克风和音频播放能力，而服务端密钥也不需要放进终端设备。

当然，PWA 不是原生 App 的完全替代品。iOS 对后台运行、锁屏通话、蓝牙按键和扬声器路由仍有限制，所以 Call Hermes 当前定位是“前台使用的个人语音入口”，而不是电话级后台通话工具。

## 现在使用起来是什么样

打开页面后，首先只会看到连接按钮。连接成功后，底部出现固定的麦克风按钮，接下来可以直接开始说话。

系统会自动判断一句话什么时候结束，不需要再次点击提交。识别到完整内容后，Hermes 开始处理，文字回复会流式出现在对话中，合成后的声音也会陆续播放。

目前已经支持：

第一，多轮对话历史。页面不再只显示最新一轮，Hermes 也会携带有限长度的上下文继续交流。

第二，播放时打断。如果 Hermes 还在说话，用户可以直接开口，系统会停止当前播放并处理新的问题。

第三，麦克风暂停。关闭麦克风后，浏览器会暂停采集有效麦克风内容，并通知服务端停止 ASR，而不是只改变界面状态。

第四，声音设置。可以选择多种男女声，调整播放速度，也可以设置停顿多久后判定“说完了”。

第五，设备和网络状态。页面能够显示手机或蓝牙输入设备，并根据 WebRTC 的 RTT、抖动和丢包情况展示网络质量。

此外还提供文字 Debug 模式。调试 ASR 或麦克风不方便时，可以直接输入文字，单独验证 Hermes 和 TTS 链路。

## 一句话背后的完整链路

用户说出一句话后，实际会经过下面这条路径：

PC 或手机浏览器通过 WebRTC 上传麦克风音频，voice-bridge 将音频统一重采样为 16kHz 单声道 PCM，然后送入阿里云百炼的实时语音识别模型。ASR 给出最终文本后，服务端调用 Hermes 的 OpenAI Chat Completions 兼容接口。Hermes 的回复增量被持续送入百炼实时 TTS，生成的 PCM 音频再转为 WebRTC 使用的 48kHz 音频轨，最终由当前设备播放。

服务端使用 FastAPI 和 aiortc。实时 ASR 采用 fun-asr-realtime，实时 TTS 默认采用 qwen3-tts-flash-realtime。公网访问使用 HTTPS，复杂网络下则由 coturn 提供 TURN 中继。

前端和服务端之间还有一条 WebRTC DataChannel，用于传输 listening、thinking、speaking、识别文本、回复增量和错误等事件。因此界面状态与音频传输可以保持同步。

## 真正花时间的不是“接通”，而是“稳定”

把几个 API 连起来并不困难，困难的是让它在移动网络和 Safari 中听起来像一次连续对话。

### 1. 语音回复为什么会断断续续

实时 TTS 返回音频的速度并不总是稳定。如果音频刚到一点就立刻播放，下一批数据稍晚几十毫秒，用户就会听到明显卡顿。

Call Hermes 在服务端维护音频队列，并根据网络质量动态调整预缓冲。网络较好时使用较短缓冲降低延迟，网络变差时增加缓冲。如果播放途中发生供给不足，会先重新积累一段音频再继续，而不是在声音和静音之间每隔几十毫秒切换。

音频发送还使用单调时钟调度。即使事件循环短暂阻塞，也不会为了追赶旧时间轴而突然发送一批过期 RTP 帧。

### 2. 为什么有时识别不到第一个字

如果检测到音量超过阈值后才启动 ASR，句首很容易已经过去。特别是爆破音或较短的中文词，第一个字可能直接被截掉。

解决方法是 VAD 前滚缓存。系统持续保留最近约 500 毫秒的音频，一旦确认用户开始说话，会把这段前导音频和后续内容一起送入 ASR。这样既不需要一直维持识别会话，也能尽量保住句首。

### 3. 支持打断后，为什么回复反而容易被截断

播放 TTS 时，手机麦克风可能再次收到扬声器里的声音。如果把任何 ASR 最终结果都当成用户打断，Hermes 的回复就会被自己的回声取消。

项目现在会区分“检测到声音”和“确认用户打断”。只有文本达到条件，并且不像正在播放的助手内容时，才会真正停止当前回复。这个判断既要避免回声误触发，又不能让用户失去自然插话的能力。

### 4. 为什么网络一直显示 Poor

浏览器提供的是累计 WebRTC 统计。如果第一次采样直接拿累计丢包计算，连接建立阶段的少量异常会被长期放大。

现在系统先建立统计基线，再使用 12 秒滚动窗口计算丢包率，同时对最差等级进行连续采样确认。网络提示因此更接近真实通话体验，也能用于调整服务端音频缓冲。

## 密钥不会进入手机

这是一个面向个人部署的项目，但“个人使用”不代表可以忽略安全。

DashScope API Key、Hermes 凭据和 TURN 密码只保存在服务端环境变量中。PWA 使用自定义共享密钥换取短期 JWT，再调用 WebRTC 信令接口。Hermes API 建议只监听 127.0.0.1 或受保护的内网地址，公网只开放 HTTPS 服务和 TURN 所需端口。

仓库也默认忽略本地 .env、TLS 私钥、日志和 PID 文件，避免在提交代码时意外上传个人配置。

## 如何部署

运行环境需要 Linux、Python 3.11、可访问的 Hermes Chat Completions API、阿里云百炼 API Key、域名和有效 TLS 证书。移动网络使用场景建议同时部署 coturn。

项目不依赖 Docker。配置好 server/.env 和 ssl 目录中的证书后，可以直接使用仓库内脚本启动 HTTPS 服务。Web 页面默认使用 10005 端口，TURN 默认使用 10004 端口，两者用途不同。

仓库提供健康检查、配置检查、TURN 检查以及自动化测试。健康接口会分别报告 Hermes、ASR、TTS、证书、TURN 和 WebRTC 缓冲配置，前端诊断日志则可以直接在设置页面查看。

## 接下来还可以做什么

Call Hermes 当前已经能够完成稳定的前台多轮语音对话，但仍有不少可以继续探索的方向：更准确的端侧 VAD、更自然的语音情绪控制、会话持久化、可选的强制 TURN 策略，以及真正的原生后台通话能力。

对我来说，这个项目最有意思的地方并不是给聊天页面加了一个麦克风，而是把本地智能体变成了一个随时可以开口交流的个人入口。

当交互从“打开电脑、找到窗口、输入命令”变成“拿起手机直接说”，Hermes 才真正开始进入日常使用场景。
